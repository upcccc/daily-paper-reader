Abstract 

Privacy-preserving aggregation is a cornerstone for AI systems that learn from distributed data with-out exposing individual records, especially in federated learning and telemetry. Existing two-server protocols (e.g., Prio and successors) set a practical baseline by validating inputs while preventing any single party from learning users’ values, but they impose symmetric costs on both servers and com-munication that scales with the per-client input dimension L. Modern learning tasks routinely involve dimensionalities L in the tens to hundreds of millions of model parameters. We present TAPAS, a two-server asymmetric private aggregation scheme that addresses these limita-tions along four dimensions: (i) no trusted setup or preprocessing, (ii) server-side communication that is independent of L (iii) post-quantum security based solely on standard lattice assumptions (LWE, SIS), and (iv) stronger robustness with identifiable abort and full malicious security for the servers. A key design choice is intentional asymmetry: one server bears the O(L) aggregation and verification work, while the other operates as a lightweight facilitator with computation independent of L. This reduces to-tal cost, enables the secondary server to run on commodity hardware, and strengthens the non-collusion assumption of the servers. One of our main contributions is a suite of new and efficient lattice-based zero-knowledge proofs; to our knowledge, we are the first to establish privacy and correctness with identifiable abort in the two-server setting. 

# 1 Introduction 

AI-driven systems increasingly rely on privacy-preserving aggregation to unlock collective insights with-out exposing individual data, most notably in federated learning and telemetry [Cor20, BKM +22, BBG +20, BIK +17]. Two-server protocols such as Prio [CGB17], Prio+ [AGJ +22], and Elsa [RSWP23] have estab-lished a practical baseline: in these designs, the two servers are symmetric, receiving and computing the same information, typically under secret sharing, so no single party learns users’ inputs, and malformed contribu-tions are validated. Prio [CGB17] is under active standardization consideration within the IETF [PBS22]. However, in modern workloads where per-client vectors are extremely large (millions of parameters) and the number of clients n is comparatively modest (often 50 − 5000 as reported in Table 2 of [KMA +21]), these symmetric designs exhibit several bottlenecks. First, each server incurs bandwidth and computation that scale linearly with the vector dimension L, so for a model of dimension L both servers must provision network throughput O(n · L) and memory O(L), effectively doubling deployment cost and forcing the sec-ondary server to match the primary aggregator’s capacity. Second, server-side communication is tied to L

rather than n, misaligning cost with the true bottleneck in federated learning and straining network and memory budgets. Third, setup or preprocessing requirements (for example, trusted setup or client precom-putation) complicate operations and introduce assumptions that reduce flexibility. Fourth, security relies on non–post-quantum, limiting long-term robustness against quantum adversaries. Finally, robustness against malicious servers is weaker: once servers begin exchanging messages, a malicious server can trigger aborts that appear indistinguishable from client misbehavior, thereby enabling blame shifting and denial-of-service 1

> arXiv:2603.19949v1 [cs.CR] 20 Mar 2026

without attributable fault. Prior work does not provide comprehensive guarantees for malicious security in server-side computation. Addressing these limitations calls for an asymmetric two-server design that removes trusted setup and preprocessing, decouples server-side communication from L so costs scale with n rather than vector di-mension, bases security solely on standard post-quantum lattice assumptions (LWE, SIS), and strengthens robustness with identifiable abort that attributes faults to misbehaving servers once inter-server messaging begins, thereby aligning private aggregation with real-world constraints in federated learning and telemetry. The recent Heli [LCGDW26] work explores asymmetric server roles but retains the aforementioned bottle-necks (setup/preprocessing overhead, communication scaling with L, non–post-quantum assumptions, and weak robustness under malicious servers). Moreover, Heli imposes a small-plaintext constraint: each vector coordinate must lie in a limited range to enable decryption via discrete-log recovery, which is impractical for high-precision or large-dimension aggregations common in federated learning. 

## 1.1 Our Contributions 

We formally introduce TAPAS , a Two-Server Asymmetric Private Aggregation Scheme that addresses these limitations along four key dimensions: 

• No setup or preprocessing: TAPAS operates without trusted setup or client-side preprocessing phases beyond the round’s standard participation, simplifying deployment and removing assumptions 

• Communication independent of vector length: TAPAS’s server-side communication is independent of the per-client vector dimension L, aligning costs with the number of participants n rather than with model size. This is crucial in federated learning, where L can be on the order of millions of parameters while n

is comparatively modest. Asymptotic costs for TAPAS and prior two-server protocols are summarized in Table 1. 

• Post-quantum foundations: TAPAS is the first two-server private aggregation protocol built solely on standard lattice assumptions (LWE, SIS), providing post-quantum security without resorting to composite assumptions or ad hoc primitives. 

• Lattice-based Zero Knowledge Proof System: Standard lattice-based proofs suffer from a soundness slack proportional to the square root of the witness dimension, √L [LNP22]. While decomposing the witness into k chunks of dimension d theoretically tightens this slack to ∝ √d, requiring simultaneous acceptance in across all blocks results in a negligible joint success probability. TAPAS combines the witness decomposition trick and an Independent Restart strategy, where each chunk runs rejection sampling asynchronously until acceptance. This ensures that the success of a block is independent of the success of other blocks, thereby significantly improving performance. This restores linear expected runtime while maintaining a tighter √d- factor slack. We prove security under a relaxed soundness definition : we guarantee that a witness 

sj can be extracted for every block j, but make no guarantee that s1 = s2 = . . . . Fortunately, for our intended application of building a two-server secure aggregation protocol, this relaxed soundness definition can be easily rectified within the overall protocol. 

• Stronger robustness with identifiable abort: TAPAS provides identifiable abort once servers begin exchang-ing messages. If a server acts maliciously, the protocol produces evidence that attributes the fault to that server (rather than “blaming” honest clients), thereby closing a gap in prior systems in which a malicious server could trigger aborts that appear indistinguishable from client misbehavior. The recent work of LZKSA [LL25] proposes a lattice-based ZK proof tailored to secure aggregation; however, we found that the protocol does not provide zero-knowledge or soundness. We defer more information in Section B. 

• Concrete performance gains: Experimental results convincingly demonstrate the significant performance gains that TAPAS achieves, a 93 × speedup over Prio and 17.5 × over Elsa for vector dimensions of L = 2 18 ,maintaining high efficiency for million-parameter models by decoupling secondary server overhead from the vector length. A central design choice that enables these advances is asymmetry between the two servers: one server per-forms lighter computation and lower-bandwidth tasks, while the other undertakes the heavier verification and aggregation work. ## 1.2 Detailed Comparison with Related Work 

Table 1 summarizes the asymptotic complexity of TAPAS against state-of-the-art protocols. The column labeled server security admits three values: MP (Malicious Privacy) indicates whether honest clients’ inputs remain private even if the servers are malicious. SH (semi-honest) denotes privacy against semi-honest servers (but malicious clients), and IA (identifiable abort) means that an honest server can detect malicious behavior and abort, revealing the identity of the misbehaving party. The final column indicates whether the protocol is robust against active client deviations, such as malformed ciphertexts or invalid plaintexts. As can be seen: TAPAS offers the highest level of security among all the other candidates. TAPAS is distinguished by its communication and computational efficiency, making it well-suited for large-scale secure aggregation. Its variants ( TAPAS BP and TAPAS LWE ) offer enhanced security and robustness, with only modest increases in 3cost, and remain competitive with or superior to existing two-server protocols in most settings. We discuss in detail below: 

Comparison with Prio [CGB17] and Prio+ [AGJ +22]: Prio-based systems employ Secret-Shared Non-Interactive Proofs (SNIPs). Clients split their data into additive shares and prove validity properties (e.g., range limits) by evaluating an arithmetic circuit over these shares. Prio+ optimizes this by performing a Boolean conversion to reduce communication. These systems impose a symmetric computational burden. To verify the SNIPs, every server must process the full vector of dimension L. Further, these works only provide privacy or guarantees against a semi-honest server. 

Comparison with Elsa [RSWP23]: Elsa employs an interactive "trust-but-verify" approach in which servers collectively verify client inputs and reject batches that fail consistency checks. Elsa achieves Malicious Privacy (MP), a property in which even if a server is malicious, an honest client’s input remains private. If a consistency check fails, the honest server cannot distinguish between a malicious client sending bad data and a malicious server falsely reporting a check failure. Elsa also does not provide malicious server security for the final output of the computation. 

Comparison with Heli [LCGDW26]: Heli is an aggregation protocol based on the Discrete Logarithm Assumption (DL). At its core, Heli builds atop the key-homomorphic PRF-based private stream aggrega-tion [SCR +11]. Heli faces two critical limitations due to its DLP foundation. First, recovering the aggregate requires solving the discrete log problem, which necessitates using small integers to ensure tractability. Sec-ond, validity proofs rely on bit-decomposition, expanding the vector size from L to L · t (where t is the bit-width). It should also be noted that a client in Heli communicates only with the heavy load server. These are the improvements that TAPAS offers over Heli: 1. Better Security: TAPAS achieves Identifiable Abort (IA) . This is a significant improvement over prior work. By enforcing a signed binding phase before aggregation, TAPAS ensures that any protocol deviation is cryptographically attributable. If the protocol halts, the honest parties output the cheater’s specific identity, thereby preventing anonymous stalling attacks. 2. Asymmetry: Unlike symmetric workload protocols such as [CGB17, AGJ +22, RSWP23], the asym-metric Heli optimizes for client count n (handling dropouts), TAPAS optimizes for model dimension L.Given that modern FL models are massive ( L ≫ n), our L-independent facilitator significantly reduces the system-wide cost in deep learning contexts. 3. Input Independence: TAPAS supports arbitrary integer inputs without decryption overhead, as lattice decryption is a linear operation rather than an exponential search. Meanwhile, Heli restricts the input domain for efficient recovery of the aggregate. 4. Proof Efficiency: While both Heli and a variant of TAPAS rely on using bulletproofs [BBB +18] as a core building block for zero-knowledge proofs, TAPAS ’s careful optimization ensures that the length of the proof vector does not incur a multiplicative t factor where t is the bit length of the inputs. By integrating Bulletproofs over lattice commitments, our proof sizes scale with O(L) rather than the bit-expanded O(tL ) required by Heli. Furthermore, we present a fully lattice-based proof system that avoids expensive group exponentiation and relies solely on field arithmetic. 

## 1.3 Technical Overview 

In this section, we outline our approach towards building TAPAS . As is typical in a two-server setting, a client communicates directly with both servers. However, in a departure from prior work, we will avoid the standard approaches of sharing the inputs between the two servers but instead rely on a careful combination of encryption and homomorphism. The protocol’s sequence diagram is described in Figure 1, proceeding in 4 phases: 

• Phase 1 (Upload): Clients distribute encrypted inputs to Server 1 ( S1) and a much shorter secret component (seed) to Server 2 ( S2). 

• Phase 2 (Consistency Check): S1 validates the received commitments against hash digests provided by S2 to detect malformed inputs. 4Figure 1: Sequence Diagram of TAPAS showing client communication with the two servers and additional rounds of communication between the two servers. The unmasked result is communicated with all the users, in the semi-honest version. The version that is secure against malicious servers also allow the participants to verify the aggregate using commitments. 

• Phase 3 (Unmasking): Upon identifying the active client set, S2 provides the necessary aggregated secret components to S1, which then computes the final plaintext result. 

• Phase 4 (Output): The unmasked aggregate is finalized in the semi-honest version. The version that is secure against malicious servers also allow the participants to verify the aggregate using commitments.

Incorporating zero-knowledge proofs. Consistent with prior work, TAPAS allows the client to generate a Zero-Knowledge Proof (ZKP) attesting to two main properties: (1) the client has executed the masking behavior correctly, and (2) the input vector xi adheres to the infinity norm bound ∥xi∥∞ ≤ t for a public value t (or the L2 norm of the witness xi). The checks on norm bounds were considered to be sufficient to mitigate poisoning attacks [NHC22,SHKR22]. To enforce these checks, TAPAS utilizes a “commit-and-prove” strategy. We offer two distinct instantiations of this protocol - one based on bulletproofs over groups of prime order where DDH is hard and the other relying on lattice-based hardness assumptions. Each variant offer a tradeoff and users can choose variants based on specific performance or security needs. We now look at the variants in greater detail. We begin by looking at the group-based approach which is a warm up to our lattice-based zero knowledge proof system. The lattice-based proof systems come with inherent challenges that we tackle in novel ways, as described later in this section. Our focus over the next two subsections (Section 1.3.1,1.3.2) are to create proof systems that protect against malicious client behavior and against a malicious server S2. We describe how we achieve privacy against malicious server S1 in Section 1.3.3 

1.3.1 Group-Based Zero Knowledge Proof Instantiation. 

This approach utilizes Bulletproofs and performs proofs over a group G of prime order q′. It guarantees the exact satisfaction of the bound ∥xi∥∞ ≤ t (i.e., there is zero slack in the proof). It is highly efficient in communication, using Pedersen commitments, a single group element can commit to an entire vector. It requires computationally expensive group exponentiations. Additionally, it necessitates a group order

This approach is designed to align more closely with the system’s underlying cryptographic assumptions. We first formally show that the LWE masking is inherently a commitment to vector xi using randomness si, ei.We then carefully build a lattice-based zero-knowledge proof specifically tailored to our intended protocol. This includes careful optimizations. All proofs are performed directly over the LWE modulus q, avoiding the need for a separate, larger group field. The proof guarantees are looser, proving only that ∥xi∥∞ ≤ γ · t,where γ > 1 represents a soundness slack, which is sufficient for most applications. A central challenge in LWE-based aggregation is proving that the noise vector e ∈ ZL is small without revealing it. Standard lattice Σ-protocols operate monolithically: the prover masks the entire vector and performs rejection sampling on the full response z. This creates a harsh trade-off: to ensure the rejection probability is non-negligible for high dimensions (for instance L ≈ 10 5), the masking noise σ must be set extremely large. This introduces a polynomial "slack" ( √L), yielding a bound that is far looser than the actual witness, thereby degrading model accuracy. TAPAS adopts a block-wise rejection sampling paradigm that decouples the protocol’s convergence from the vector dimension L.

• Decomposition & Independent Restarts: We partition the witness into small blocks of size 

d ≪ L. Crucially, we treat the rejection condition for each block independently. If a specific block fails the rejection check (i.e., falls into the tail of the distribution), the prover restarts the protocol 

only for that block , leaving successful blocks untouched. This transforms the convergence time from exponential in L to linear in L, allowing us to target significantly tighter distributions. It is to be noted that this decomposition and independent parallel execution of Σ could allow a malicious client to use a different sj , per block. However, for our intended application to building a two-server secure aggregation protocol, we have built in additional checks to remedy this attack. 

• Bimodal Gaussian Targeting: Leveraging the independent restart policy, we employ Bimodal Rejec-tion Sampling [DDLL13]. By targeting a distribution that matches the shape of the witness-plus-noise, we reduce the required standard deviation σ.

• Fixing LZKSA: We note that the recent LZKSA protocol [LL25] attempts a similar efficiency gain but fails to achieve soundness. LZKSA permits norm checks to wrap around modulo q, allowing adversaries to hide unbounded inputs. TAPAS enforces strict integer-based checks over the decomposed blocks, providing the first sound realization of this high-efficiency proof strategy. 1.3.3 Security Against Malicious Server S1.

Proving privacy against a malicious server typically requires the simulator to manipulate the aggregate sum. This is a feature commonly observed in single-server protocols [BBG +20, MWA +23, KP25, BCGL +25] where this is achieved by adding a secondary mask produced by a function H, modeled as a programmable random oracle. The simulator waits until the adversary chooses the corrupt set, then programs H(x) to inject a specific correction term into the sum. The core philosophy behind this approach is to ensure that the communication is independent of the vector length as |x| ≪ L where L is the length of the vector being masked. This same approach can be undertaken for our asymmetric two-server case for malicious S1. For our protocol, the new masking function is y = As + e + ∆ x + H′(seed ) for some seed that is expanded by H′.The client sends seed to S2. Once the online set is fixed, S2 sends individual seeds to S1. S1 now applies H′

on each seed seed to remove from the sum of the ciphertexts. If S1 is malicious, the simulator waits until the online set is fixed to query the functionality and then correct, with the help of random oracle programming, to ensure that the sum of ciphertexts decrypts to the actual result expected by the adversary controlling 

S1. However, this approach of using a second mask appears to collapse when clients are required to prove correctness of their inputs where the witness now includes seed and the proof π is to show that y is correctly formed which involves proof of correct computation of H′(seed ). Implicitly, it appears to ask for a proof π

to show that there is a witness x such that π = H(x). Leaving aside the fact that zero-knowledge proofs to prove correct computation of hash function [GMO16] are expensive, in our case after S1 has verified π, it receives both the input x and the newly programmed value y′ with the expectation that π verifies even given witness x and y′. Typically, π contains a binding commitment to y which might fail with y′. Prior works on single-server secure aggregation with input validation [BGL +23,MGKP25] have constructions only based on security against semi-honest server. In this work, we circumvent this barrier by introducing an equivocable commitment scheme, restoring the simulator’s ability to program the oracle consistently. # 2 Preliminaries 

We consider a federated learning framework. There exist n clients, with each client C(i) owning a dataset 

D(i). The server holds the ML model Θ. In FL, the server first sends Θ to the clients, and each client trains on its local dataset D(i) to generate the updated weights m(i). Meanwhile, the server computes the average of these model updates m(i) to update its global model to Θ′. In the next iteration, Θ′ is sent back for the next update. The goal is to enable the server to compute the average and, thereby, the new model Θ′, while keeping the model weights confidential. To this end, we employ an intermediary party, the second server. 

Communication and Threat Model. TAPAS operates within the traditional two-server setting with n clients [RSWP23, CGB17, AGJ +22], where the threat model assumes that the two servers cannot collude. We designate these as servers S1 and S2. In contrast to prior two-server protocols, our second server S2 performs a lighter computational load than server S1, introducing an asymmetry in the server roles. Additionally, unlike tradi-tional two-server protocols where both servers receive client inputs proportional to the input length, TAPAS 

reduces the communication received by S2 to be independent of the input length. 

Modeling Security. Like prior works, we identify two kinds of behavior. A semi-honest party is one that adheres to the protocol description but remains curious and attempts to compromise the inputs of the honest parties. A malicious/corrupted party is one that is allowed to arbitrarily deviate from the protocol description. TAPAS is designed to offer the following feature: honest clients’ inputs remain private as long as at least one server is honest and does not collude with the other server. Furthermore, once the servers begin interacting, we guarantee an identifiable abort, i.e., malicious clients (or the server) are detected.

## 2.3 Simulation-Based Proof of Security 

Our proof follows the standard simulation-based paradigm [Gol04, Lin17]. The objective is to show that any real-world adversary attacking our protocol can be simulated by an adversary S in an ideal world , where a 

trusted party T securely computes a function F on the clients’ behalf. In our setting, this function simply aggregates (sums) the clients’ input vectors. We consider an adversary A that can corrupt up to η · n clients and at most one server. The ideal-world execution proceeds as follows, specialized to the case where only the server receives the final output: (a) Each honest client submits its input to the trusted party T .(b) The simulator S determines which corrupted clients provide valid inputs and which ones abort. (c) If a server is corrupted, S may decide either to abort the execution or to allow it to proceed. (d) If no abort occurs, T computes the function F (X) and sends the result to the server. (e) If the server is honest, it outputs whatever value it receives from T .We formally define the functionality in Figure 3. We allow for the following adversarial behavior: 

• Clients can be malicious, i.e., deviate from protocol descriptions arbitrarily. 

• We require exactly one server to be honest, and the other server to be adversarially controlled. Looking ahead, we model this adversarial control to be either semi-honest or malicious and still offer privacy. Observe that a malicious server can “forge” or provide inputs on behalf of the honest clients. To avoid this trivial attack, we assume that clients’ messages are signed. We assume that there exists a Public Key Infrastructure (PKI) that allows servers to verify and validate signatures. Later, we additionally leverage these signatures to achieve security with identifiable abort.

3.2.2 Security Against Malicious Servers. 

Before we present the malicious version of the protocol it is necessary to discuss the additional properties needed in this version. They are: 

• As shown in the malicious version of TAPAS BP , we need a secondary mask w(i) that is the output of a hash function H′ on seed (i). The purpose of this is two fold: (a) use this to program the random oracle model once the online set is finalized by server S1, and (b) to ensure that the communication from S2

to S1 is independent of the vector length. 

• Similar to the other malicious version, we need server S2 to prove to S1 that it has provided the correct 

sagg , with respect to the online set. Unfortunately, with the information provided to S1, it cannot be convinced. To this end, we need a second commitment that is additively homomorphic, lattice-based and compatible with the rest of the proof protocol. 

• Finally, we need an equivocable commitment. This is critical because while we provide a commitment to w(i) and prove that y(i) is correctly computed with respect to this commitment, S1 needs to be convinced that indeed the seed (i) provided by S2 when expanded opens the commitment to w(i). We 19 need this level of equivocation. To this end we will use the Ajtai commitment [Ajt96, LNP22]. A key artifact that will be useful for our protocol is that the dimension of the message s1 does not increase the commitment size. Further, since SIS hardness is fairly independent of the size of the solution, increasing the length of s1 does not impact the security. However, we do require that the norm of s1 is bounded. ## 3.3 Malicious Security with Identifiable Abort 

Existing efficient aggregation protocols, such as Elsa [RSWP23], typically achieve only Security with Abort 

(SwA). In these systems, if a server detects an inconsistency (e.g., a checksum failure), it must abort to preserve privacy, but it cannot cryptographically prove who caused the failure. This creates a “framing” vulnerability: a malicious server can falsely report a failure to halt the protocol, effectively censoring honest clients without being identified. We demonstrate that TAPAS achieves the stronger property of Security with Identifiable Abort (IA) by incorporating digital signatures. This ensures that any abort is accompanied by cryptographic evidence identifying the cheater (whether a client or a server). 

Ideal Functionalities: SwA vs. IA. Let n be the number of clients and A be the adversary controlling a subset of corrupt clients C ⊂ [n].

• Standard Security with Abort ( FSwA ): If A sends a malformed input, the functionality outputs 

⊥ to all parties. Deficiency: Honest parties learn that someone cheated, but learn nothing about the identity of the malicious party. 

• Identifiable Abort ( FIA-Agg ): If the functionality detects an invalid input or protocol violation from client j ∈ C (resp. from server Si), it halts and outputs (⊥, j ) (resp. (⊥, S i) Advantage: The server 

S1−i (and other honest parties) obtains the identity. If it is client j, then the servers can evict the malicious client and then restart the protocol. 

The Mechanism. The core intuition is to enforce a Binding Agreement Phase before aggregation. Clients sign their inputs, and servers must commit to the set of valid signatures they have received before processing data. This prevents a malicious server from initially accepting a client’s data and later claiming it was 21 malformed to force an abort. Let Σ = ( KeyGen , Sign , Verify ) be an EU-CMA secure signature scheme. The protocol proceeds as follows: 1. Authenticated Submission. Client Ci signs their payload. For a message mi,ℓ destined for Server ℓ,the client sends ¯mi,ℓ = ( mi,ℓ , σ i,ℓ ), where σi,ℓ ← Sign (sk i, m i,ℓ || ℓ).2. The Handshake (Binding Agreement). Before exchanging any user data, the servers exchange their sets of locally valid clients, V1 and V2.

• They compute the intersection V = V1 ∩ V 2.

• Critical Step: Each server signs and exchanges the set V.By signing V, Server ℓ explicitly attests: “I possess a validly signed message from every client in V.” 3. Blame Attribution. During aggregation, if a consistency check fails for Client i ∈ V , the servers enter a Blame Phase by exchanging the signed transcripts (mi, 1, σ i, 1) and (mi, 2, σ i, 2).

• Case A: Malicious Client (Equivocation). If both signatures σi, 1, σ i, 2 are valid but the messages are inconsistent (e.g., mi, 1 and mi, 2 do not sum to a valid vector), Client i is identified as malicious. The signed conflicting messages serve as proof of misbehavior. 

• Case B: Malicious Server (Framing/Forgery). If Server 1 claims an error but cannot produce a valid signature σi, 1 for Client i, Server 1 is identified as malicious (having violated the Handshake agreement). Similarly, if Server 1 provides a message that is actually consistent with Server 2’s message (contradicting the error claim), Server 1 is identified as malicious for raising a false alarm. # 5 Conclusion 

In this work, we presented TAPAS , a novel two-server secure aggregation protocol that fundamentally redefines the resource allocation in privacy-preserving federated learning. By enforcing an asymmetric de-sign, TAPAS decouples the communication and computation costs of the secondary server from the vector dimension L and scales instead with the number of clients n. This addresses a critical scalability bottle-neck in prior symmetric protocols such as Prio and Elsa, making TAPAS uniquely suitable for aggregating high-dimensional models (e.g., deep learning parameters) where L ≫ n.Our construction leverages standard lattice-based assumptions (LWE and SIS), ensuring post-quantum security without relying on trusted setups or pre-processing. We introduced a tailored Zero-Knowledge proof system that uses block-wise rejection sampling, which mitigates the soundness slack typically associated with lattice proofs while maintaining efficient runtime. Furthermore, TAPAS achieves Identifiable Abort and full malicious security, ensuring that protocol deviations are cryptographically attributable to the malicious party, whether client or server. Experimental microbenchmarks suggest that TAPAS offers an efficient path for deploying secure ag-gregation at scale, combining the rigorous security guarantees of lattice cryptography with the efficiency required for modern machine-learning workloads. Future work may focus on further optimizing the con-stants in the lattice-based proof system and exploring client-side compression techniques compatible with LWE encapsulation.

# B Comparison with Prior Work [LL25] 

Recent work by [LL25] proposed a secure aggregation scheme for federated learning which uses lattice-based proof systems. Unfortunately, their constructions have security gaps. To begin with, the defined commitment scheme cannot be binding, because the protocol validates the commitment relation without enforcing constraints on randomness, allowing a malicious prover to alge-braically compute a noise term that effectively opens the commitment to any arbitrary message. Although the authors specify distributions ostensibly to address these concerns, they are never formally defined in the remainder of the paper. As shown in Lemma 2, we eliminate this vulnerability by grounding binding strictly in the hardness of the Short Integer Solution (SIS) problem. Unlike LZKSA, our protocol enforces computational binding by requiring a Zero-Knowledge Proof of Knowledge (ZKPoK) for the randomness vectors s, e with strict norm bounds when the commitments are used later to build zero-knowledge proofs, a feature missing from LZKSA. 

Background: Σ-Protocols in DL vs. Lattices. In classical discrete-logarithm protocols (e.g., Schnorr), the response z = r + cx (mod q) is reduced modulo q. This modular reduction perfectly hides the witness x

(ensuring Zero-Knowledge) and allows the challenge c to be drawn from the full field Zq (ensuring negligible soundness error). In contrast, lattice-based arguments rely on the witness having a small norm. Consequently, the response 

z = r + cs must be computed over the integers (Z), not modulo q. This introduces two fatal constraints that LZKSA ignores: 32 1. Small Challenge Space: To prevent norm explosion, c must be small (e.g., binary), which yields a high soundness error (1/2) in a single execution. 2. Leakage via Distribution: Without modular reduction, the distribution of z depends on s. Standard constructions require Rejection Sampling ( [LNP22,DDLL13,Lyu12,Lyu08]) to decouple z from s. This is done by sampling r from a much larger space so that it “floods” cs.A rigorous inspection of the LZKSA protocol [LL25] reveals that it fails to satisfy the fundamental re-quirements of lattice-based Zero-Knowledge arguments. We identify two fatal flaws in their adaptation of discrete-logarithm Σ-protocols to the lattice setting: 

• Soundness: LZKSA leaves the challenge space C undefined. If C is small (to allow extraction), a single execution yields a non-negligible cheating probability of 1/2. If C is large (to ensure soundness), the extracted witness norm explodes, preventing a valid reduction to SIS. LZKSA performs only a single execution, rendering the proofs unsound. 

• Zero-Knowledge: LZKSA omits Rejection Sampling entirely, outputting the raw vector z = r + cs.Consequently, the conditional distribution of z shifts by s. A verifier can recover the secret witness s by simply averaging the difference between the response and the expected noise distribution, completely breaking the privacy of the aggregation scheme. 

TAPAS resolves these issues by implementing a fully rigorous lattice ZK argument. We fix the challenge space to C = {− 1, 0, 1} and perform parallel repetitions to drive the soundness error below 2−λ. Furthermore, we employ Bimodal Rejection Sampling [DDLL13] to ensure the response distribution is statistically independent of the witness, guaranteeing Zero-Knowledge while maintaining compact parameters.
