## 摘要
无服务器平台上的联邦学习（FL）聚合面临着严峻的可扩展性瓶颈：现有架构（如 lambda-FL、LIFL）在聚合器之间划分客户端，但每个聚合器都必须在内存中持有完整的模型梯度。当梯度超过单个函数的内存限制（例如 AWS Lambda 上的 10 GB）时，无论树深度或分支因子如何，聚合都变得不可行。我们提出了 GradsSharding，它将梯度张量划分为 M 个分片，每个分片由一个无服务器函数独立进行平均，该函数接收来自所有客户端的贡献。由于 FedAvg 平均是逐元素进行的，因此这会产生与基于树的方法位一致（bit-identical）的结果，从而使模型准确率在设计上保持不变。每个函数的内存占用被限制在 O(|θ|/M)，与客户端数量无关，从而能够聚合任意大的模型。我们通过 HPC 实验和跨越 43 MB 到 5 GB 模型规模的真实 AWS Lambda 部署，对比评估了 GradsSharding 与 lambda-FL 及 LIFL。结果显示，在梯度大小约为 500 MB 时出现成本交叉点，在 VGG-16 规模下成本降低了 2.7 倍，并且 GradsSharding 是唯一在超出无服务器内存上限后仍可部署的架构。

## Abstract
Federated learning (FL) aggregation on serverless platforms faces a hard scalability ceiling: existing architectures (lambda-FL, LIFL) partition clients across aggregators, but every aggregator must hold the complete model gradient in memory. When gradients exceed the per-function memory limit (e.g., 10 GB on AWS Lambda), aggregation becomes infeasible regardless of tree depth or branching factor. We propose GradsSharding, which instead partitions the gradient tensor into M shards, each averaged independently by a serverless function that receives contributions from all clients. Because FedAvg averaging is element-wise, this produces bit-identical results to tree-based approaches, so model accuracy is invariant by construction. Per-function memory is bounded at O(|θ|/M), independent of client count, enabling aggregation of arbitrarily large models. We evaluate GradsSharding against lambda-FL and LIFL through HPC experiments and real AWS Lambda deployments across model sizes from 43 MB to 5 GB. Results show a cost crossover at approximately 500 MB gradient size, 2.7x cost reduction at VGG-16 scale, and that GradsSharding is the only architecture that remains deployable beyond the serverless memory ceiling.

*   **核心问题**：在无服务器（Serverless）架构下进行联邦学习（FL）聚合时，存在严重的**内存瓶颈**。现有的架构（如 $\lambda$-FL 和 LIFL）通过在多个聚合器之间划分客户端来扩展规模，但每个聚合器仍需在内存中持有**完整的模型梯度**。
*   **研究背景**：随着模型规模从数亿参数增长到数百亿，梯度张量的大小往往会超过无服务器函数（如 AWS Lambda）的单实例内存上限（通常为 10 GB）。一旦模型超过此限制，现有的树状聚合方案无论如何增加深度或分支都将失效。
*   **动机**：利用无服务器计算的按需计费和自动扩展特性，解决大模型聚合时的内存溢出问题，并降低因参数服务器（PS）长时间闲置导致的资源浪费。

### 2. 论文提出的方法论：GradsSharding
*   **核心思想**：将梯度张量的**维度**（而非客户端数量）进行切分。将完整的梯度划分为 $M$ 个分片（Shards），每个分片由一个独立的无服务器函数负责聚合。
*   **关键技术细节**：
    1.  **分片（Shard）**：客户端将本地训练产生的梯度向量 $g_i$ 切分为 $M$ 个连续分片。
    2.  **上传（Upload）**：客户端将 $M$ 个分片并行上传至 S3 存储桶。
    3.  **聚合（Aggregate）**：触发 $M$ 个并发的 Lambda 函数。每个函数 $j$ 从所有 $N$ 个客户端下载第 $j$ 个分片，计算逐元素平均值，并将结果写回 S3。
    4.  **重构（Reconstruct）**：客户端下载所有聚合后的分片并拼接，恢复出完整的全局梯度。
*   **算法特性**：由于 FedAvg 聚合是逐元素进行的，分片聚合的结果与整体聚合在数学上完全一致（Bit-identical），因此**不影响模型精度和收敛性**。
*   **内存优势**：单函数内存需求从 $O(|\theta|)$ 降至 $O(|\theta|/M)$，实现了内存占用与模型总大小的解耦。

### 6. 主要结论与发现
*   **成本交叉点**：当梯度大小超过约 **500 MB** 时，GradsSharding 的成本开始低于树状架构。在 VGG-16 规模下，成本降低了 **2.7 倍**。
*   **性能加速**：通过并发执行 $M$ 个分片聚合，在 $M=16$ 时实现了 **16.2 倍** 的延迟加速。
*   **突破瓶颈**：当梯度超过约 **3 GB** 时，$\lambda$-FL 和 LIFL 因触发 Lambda 内存上限而无法运行，而 GradsSharding 是唯一能处理 5 GB 甚至更大模型的架构。
*   **I/O 瓶颈**：在 Lambda 聚合中，S3 读取时间占总执行时间的 **91%~99%**，计算开销几乎可以忽略不计。

### 7. 优点
*   **极高的可扩展性**：通过调整分片数 $M$，理论上可以聚合任意大小的模型，彻底解决了无服务器平台的内存硬上限问题。
*   **成本效益**：在大模型场景下，通过减少单实例内存分配显著降低了 Lambda 的计费成本。
*   **精度无损**：设计保证了聚合结果的一致性，无需担心分布式训练中的精度损失。

### 8. 不足与局限
*   **S3 请求开销**：分片增加了 S3 的 PUT/GET 请求数量。对于小型模型，S3 的请求费用可能超过 Lambda 的计算节省，导致小模型下成本反而更高。
*   **网络依赖**：高度依赖云存储的带宽。实验显示单流 S3 读取速度限制在 45-58 MB/s，这成为了主要的延迟瓶颈。
*   **客户端负担**：客户端需要处理分片上传和下载后的重构，对于计算/带宽受限的边缘设备可能增加额外负担。
*   **实验规模限制**：跨架构对比中客户端数量固定为 $N=20$，虽然内存边界不受 $N$ 影响，但更大规模客户端下的 S3 竞争情况未深入探讨。
